# DECISIONS

One line per decision: what, why.

- Product name stays "prosetype" as a placeholder pending the human answer to plan §13.1.
- CI is GitLab CI flavor since the team tooling is GitLab (plan §13.2).
- Russian originals deferred to Phase 3 per plan §13.3 (Cyrillic/IME complexity would blur the Phase 1 feel gate).
- Node 24 LTS chosen (plan §3 allows it); pinned in `.nvmrc`.
- TypeScript pinned to ~5.9 (not 6.0.3 latest): TS 6.0 shipped after this scaffold's toolchain baseline and typescript-eslint support tops out at <6.1; 5.9 is the boring, fully supported choice.
- ESLint 10 + typescript-eslint 8 flat config uses the non-type-checked `recommended` preset at the root for speed and simplicity; packages can opt into typed linting later if needed.
- Base tsconfig uses `module: ESNext` + `moduleResolution: bundler`; `apps/api` may override to NodeNext when it grows a real server build.
- Root `ingest` script runs `node scripts/ingest.ts` relying on Node 24 native type stripping; revisit if the ingest script needs non-erasable TS syntax.
- CI installs pnpm via `npm install -g pnpm@11.7.0` instead of corepack, since corepack is being removed from Node distributions.
- Stub packages each carry their own `typescript`/`vitest` devDependencies (same versions as root) because pnpm does not expose root binaries to workspace packages.
- pnpm 11's supply-chain policy (`minimumReleaseAge`) auto-appended `minimumReleaseAgeExclude` entries to `pnpm-workspace.yaml` during install for very fresh vitest/typescript-eslint releases; kept as generated.
- Fonts self-hosted via `@fontsource` packages (`ibm-plex-mono` 400/500, `eb-garamond` 400-italic) - boring, versioned, no CDN dependency (plan §9.4 silent on delivery mechanism).
- Vite dev server on 5173 proxies `/api` → `http://localhost:3001` so the web app can call the API same-origin in dev (plan silent on dev proxy).
- react-router pinned to ^7 (library mode: `BrowserRouter`/`Routes`) and vite to ^7 with `@vitejs/plugin-react` ^5, not the days-old router v8 / vite 8 (plugin-react 6 peer-requires vite 8); stable, boring combo.
- Letterbox bars use `--color-bar: #0B0906`, a touch darker than the stage, since plan §9.4 names no bar color ("never pure #000" is read as a stage rule but respected for bars too).
- Top letterbox bar carries quiet `stats · library` nav links in subtitle style - plan is silent on how to reach the other two routes.
- `allowBuilds: esbuild: true` added to `pnpm-workspace.yaml` so vite's esbuild dependency can run its postinstall under pnpm 11's build-script gate.
- zod v4 (current major) used in `packages/schema` and api config; plan pins no version, v4 is current (`z.uuid()`, `z.int()`, `z.url()` top-level APIs).
- Passage DTO (plan silent on exact shape) nests attribution as `work {slug,title,translator,pubYear}` and `author {slug,name,era}` so the client can render the epigraph without extra requests.
- charEvents 64KB cap measured as UTF-8 bytes of `JSON.stringify(log)` via `TextEncoder` (works in browser and Node).
- `/healthz` registered at `/api/v1/healthz` since plan §8 puts the whole route table under base path `/api/v1`.
- Drizzle numeric columns use `mode: 'number'` so stats read/write as JS numbers instead of strings.
- api and schema packages use explicit `.ts` import extensions + `allowImportingTsExtensions` so Node 24 native type stripping runs `src/server.ts` directly (no build step, no tsx); dev script uses `node --watch --env-file-if-exists=../../.env`.
- @fastify/cors configured with static string origin: it always answers with the configured origin and never reflects a foreign one (browser enforces); test asserts non-reflection.
- Initial Drizzle migration generated as `apps/api/drizzle/0000_init.sql` (`drizzle-kit generate --name init`); no DB connection needed for generate.
- corpus: Anna Karenina on PG (#1399) is the Constance Garnett translation, not Maude as plan §5 assumed - recorded the real PG translator.
- corpus: plan said Meditations (Long), but PG #2680 is the archaic Casaubon version - sourced from PG #15877 (George Long) instead.
- corpus: Red Harvest, As I Lay Dying, and Kafka's The Castle are not on Project Gutenberg (checked via Gutendex 2026-07-07) - substituted a second Maltese Falcon passage and extra Fitzgerald/Woolf passages; details in corpus/passages.yaml header.
- corpus: in the Maltese Falcon "blond satan" excerpt, PG typographic markup only (lead-in small caps, italic underscores around "motif") was dropped; wording untouched.
- corpus: pub_year recorded as first-publication year of the translation (or of the original for English-language works), best-effort from standard bibliography.
- corpus: era slugs beyond plan examples chosen plainly: philosophy (Nietzsche), stoic (Aurelius, Epictetus), hardboiled (Hammett), gothic (Poe), victorian (Wilde); Conrad classed modernist with hardboiled theme per §5's proto-noir framing.
- `scripts/` is a pnpm workspace package named "scripts" (added to `pnpm-workspace.yaml`) so its yaml/zod deps + vitest wiring live with the code; `ingest.ts` imports the api drizzle schema/client via relative `../apps/api/src/db/*.ts` paths rather than adding an exports map to the api package.
- Root `ingest` script gained `--env-file-if-exists=.env` so DATABASE_URL loads from `.env` like the api dev script; an optional CLI arg (`pnpm ingest path/to.yaml`) overrides the corpus path for testing (plan silent on both).
- Accent folding (§6.3) generalizes the listed examples via NFD decomposition + combining-mark strip plus a small ligature map (æ œ Æ Œ ß ø Ø); any non-ASCII char with no fold fails loudly with its code point.
- Difficulty (§6.4) definitions where the plan is silent: word length counts letters/digits only (punctuation weight is already the separate term), punctuation-per-100-chars uses total text length incl. spaces, sentence count = number of `.!?` runs (min 1).
- Passage upsert uses `onConflictDoUpdate` on `text_hash` so re-ingest refreshes computed fields idempotently; duplicate hashes within one YAML file are skipped and named in the report; author/work metadata takes the first YAML occurrence per run.
- `corpus/passages.yaml` entries are validated with `z.strictObject` so unknown keys (curator typos like `band_overide`) fail loudly; optional `birth_year`/`death_year` keys are accepted to feed the authors table.
- Ingest validates and normalizes every passage before any DB write, then upserts everything in a single transaction - a bad passage means zero writes.
- Integrator: recreated the local Postgres volume and applied schema via `pnpm db:migrate` (drizzle-kit) instead of the earlier manual psql apply, so the drizzle migration journal exists and re-running `db:migrate` is a no-op.
- Integrator: ran Prettier over `corpus/passages.yaml` (double→single quote style only; passage text blocks untouched, hashes unchanged, re-ingest confirmed idempotent at 30 passages).
- Integrator: `curation-report.txt` (output of `pnpm ingest`) committed at the repo root so the Gate 0 reviewer can read the difficulty/band report without running ingest.
- Difficulty weights calibrated and frozen (2026-07-07) per §6.4's calibrate-then-freeze: §6.4's starting weights (2.0/2.5/0.4/0.2) compressed the 30-passage seed corpus to 17.6–38.4 (26/4/0/0 across bands) because avg word length barely varies in real prose while sentence length/punctuation were under-weighted; frozen weights are 1.5·avgWordLength + 3.0·punctPer100 + 0.5·pctWords8Plus + 0.9·avgSentenceWords (band thresholds unchanged - weights were the single calibration lever), giving warmup 5 / standard 14 / hard 7 / brutal 4 with aphorisms in warmup and the Woolf/Poe/Gogol/Turgenev long-clause passages in brutal.
- Passage db access lives behind a `PassageRepository` interface (`apps/api/src/passages/`); `buildApp` accepts an optional `passageRepo` dep so route unit tests stub it (no Postgres in CI until Phase 2).
- `GET /passages/next` random selection uses `ORDER BY random() LIMIT 1` - O(n) scan, fine at the ~30-row corpus; revisit if the corpus grows large.
- `exclude` query param over the 20-id cap is rejected with 400 (not silently truncated) - a client sending more is a bug worth surfacing.
- API error bodies are `{ error: 'BadRequest'|'NotFound', message }`; 400 messages come from zod v4 `z.prettifyError` (plan silent on error shape).
- Passage repository rows are re-parsed through the shared `passageSchema` before returning so DB/DTO drift fails loudly at the boundary.
- engine depends on `@prosetype/schema` as a type-only devDependency (CharEvent/CharEvents/CharEventCodeValue), keeping schema the single source of truth for the wire format while the engine stays zero-runtime-dependency.
- engine defines its own `RunStats` interface (structurally identical to schema's) since importing the zod-derived type would pull a runtime schema dependency for a non-charEvents shape.
- engine event-index conventions where §7.5 is silent: slot adds use the target char's passage index; extras, over-cap presses, and space commits use the word-end (space) index; backspacing into the previous word logs a delete at that word's space index; all indices stay within the passage.
- over-cap keypresses (beyond +8 extras) are logged as code-1 events so replay reproduces accuracy, but they add no char: counted as incorrect keypresses, excluded from rawWpm and per-second buckets.
- accuracy counts every keypress (retypes included; "first-attempt basis" means the original miss is never erased); a space keypress is correct iff it commits a fully correct word (Monkeytype `incrementAccuracy(isWordCorrect)`); backspaces are never keypresses.
- kogasa/stdDev/mean verified verbatim against monkeytype `packages/util/src/numbers.ts` (2026-07-07): population stddev, `100*(1-tanh(cov+cov^3/3+cov^5/5))` - no divergence.
- consistency buckets (plan silent on binning): `max(1, ceil(durationMs/1000))` buckets, events at `floor(t/1000)` clamped into the last bucket, bucket raw wpm = chars*12, partial last second not scaled.
- engine stats are rounded to 2 decimals (Monkeytype-style, and what §7.4's "exact to 2 decimals" tests pin); mid-run wpm counts only committed fully-correct words (the in-progress word counts only as the last word at completion).
- space on the last word and space on an empty word are no-ops (nothing logged, not keypresses; plan silent - quote mode has no trailing space and Monkeytype ignores empty-word space).
- Ctrl/Alt+Backspace emits one code-2 delete per cleared char (same timestamp) so §7.5 needs no extra code; on an empty word it falls back to the single-step cross-into-previous rule.
- engine timestamps: caller clock (`performance.now()`) normalized to integer ms since first keystroke via `Math.round`, clamped monotonic non-decreasing (a backwards clock can never corrupt the log).
- engine does not enforce the 6000-event/64KB caps while typing (correctness first); the caps are wire-level and enforced by the shared zod schema at the submission boundary.
- web tsconfig adds `allowImportingTsExtensions` (like api/schema) since it typechecks the imported engine/schema TS sources, which use explicit `.ts` extensions.
- char-state styling where §9.4 is silent: corrected = bone with a dim smoke underline, missed = smoke with a blood underline, extras = blood at 70% opacity; incorrect slots render the target character in blood (Monkeytype default) - the typed character lives only in engine state/log.
- 3-line window (§9.3): fixed height of 3 line-boxes; the active line is pinned to the middle line once past the top, scroll measured from char span offsetTop and applied as a translateY on the words container (200ms ease); under prefers-reduced-motion the slide becomes a cut with a brief WAAPI opacity-only fade (opacity is motion-safe; CSS transitions are globally zeroed there).
- non-BMP input (surrogate pairs, e.g. emoji) is skipped by the store: the engine wire format is one UTF-16 char and such input can never be correct against an ASCII passage.
- caps-lock state lives in the typing zustand store so the shared letterbox bottom bar can render the §9.3 tag without prop plumbing.
- Tab/Esc are handled at document level (capture) while `/` is mounted, so Tab is always "next passage" there (including from the result and error views) and never moves focus - header nav on that route is mouse-only, per §9.3's "Tab never moves focus".
- client exclude list: a passage id is appended on successful load (moved to the end if repeated), capped at 20 to mirror the API's `MAX_EXCLUDE_IDS`.
- esc-restart marks the run `restarted` only if the abandoned run had actually started (first keystroke seen); restarting an untouched passage is not a restart.
- ResultView seam (`apps/web/src/result/ResultView.tsx`): exports `ResultViewProps { run: CompletedRun; passage: Passage; onNext: () => void }` with `CompletedRun { stats: RunStats; log: CharEvents; restarted: boolean }`; the Phase 1 result-view work replaces the placeholder body keeping those props.
- stage input listeners are attached natively (addEventListener) rather than via React synthetic props: `beforeinput` needs `inputType` + a cancelable event, which React's onBeforeInput does not guarantee.
- result view adds no Tab keydown listener of its own: the stage's document-level capture handler stays mounted beneath the result view and already maps Tab to loadNext there; a second listener would double-fire.
- sparkline (plan says only "minimal axes, dim smoke"): y-domain anchored 0..peak raw wpm (pauses read as honest dips); axes = smoke zero baseline, dashed peak gridline labeled with its value, 0:00/duration time labels; one tungsten 2px polyline, no legend or tooltip (single series).
- heatmap rendering (§7.6 ramp, plan silent on the exact mapping): heat h → `color-mix(in srgb, tungsten h·100%, smoke)` at opacity 0.7+0.3h; error-touched chars are blood with a thin blood underline (mark never color-alone); never-typed chars dim smoke; per-char latency/error counts surface as native `title` tooltips.
- reader-stat formatting: slowest-word times as whole milliseconds; punctuation tax rounded to a whole percent with explicit sign ("+38%"), em-dash when unsampled.
- heatmap (§7.6) definitions where the plan is silent: per-char latency = first-attempt interval from the previous keypress (deletes excluded from the chain, extras/over-cap advance it but get no sample); heat = `log1p(min(latency, p95))/log1p(p95)`; error touches = incorrect keypresses at an index (extras/over-cap land on the space index, early spaces on the space index); slowest words sum their own chars' latencies (surrounding spaces excluded); punctuation tax = mean punctuation latency vs mean letter/digit latency − 100, null when unsampled.
- Phase 2 CI host is GitHub Actions (origin is github.com/Abulkosim/prosetype); the Phase 0 `.gitlab-ci.yml` was retired - the human confirmed GitLab is not used at all, superseding the earlier §13.2 GitLab assumption.
- GET /profiles/:id/stats DTO (`profileStatsSchema`, plan silent on exact shape): `totals{tests,timeTypedMs}`, nullable `bestWpm{wpm,passageId,workTitle,authorName}`, nullable `avgWpmLast10/avgAccuracy/avgConsistency/punctuationTaxAvgPct`, `perAuthor[]`, and `history[]` (last 50, each an enriched result summary with band + work title + author). Every aggregate nullable so a zero-result profile renders cleanly.
- Stats aggregation split: all-time figures (totals, best, per-author avg, accuracy/consistency averages) come from SQL aggregates over every result; the last-10 wpm average and the punctuation-tax average are derived from the recent-50 window (`buildProfileStats`, kept DB-free and unit-tested). Punctuation tax is recomputed per recent result from its stored charEvents via `computeHeatmap`, averaging the non-null samples - bounds the engine work at 50 replays per stats read.
- POST /results returns 201; POST /profiles returns 201 (creates). `client_match` requires all four stats (wpm, rawWpm, accuracy, consistency) within tolerance `max(1.0 abs, 2% relative)` (plan §8's wpm-centric tolerance applied uniformly). Mismatches are stored and flagged, never rejected (§8).
- POST /results sanity order: zod parse (covers monotonic + 64KB + shape) → profile exists (404) → passage exists (404) → event-count plausibility (`> charCount*20+50` → 400) → engine recompute (MalformedLogError → 400) → duration < 3000ms → 400 → wpm > 350 → 400 → insert with server-computed values.
- POST /results rate-limited to 20/min via a per-route `config.rateLimit` override (tighter than the global 100/min), per §8.
- GET /authors and GET /themes live on the `PassageRepository` (one catalog owner); `listThemes` unnests the `themes` text[] via a raw `db.execute` since Drizzle has no array-unnest builder.
- api build.ts resolves each repo as `deps.X ?? drizzle(db())`, creating the Postgres client only if any repo was left to default (all stubbed in unit tests) - a `db()` thunk keeps this type-safe without non-null assertions.
- API integration tests (`test/integration.test.ts`) probe the DB at load via top-level await and `describe.skip` when unreachable, so `pnpm test` stays green on a DB-less machine while CI (Postgres service) always runs them; they self-seed distinctive `zz-it-*` fixtures and clean up, never depending on the corpus.
- Local dev DB provisioned in the running Homebrew Postgres 14 (no Node 24 available this session to run the Node-24-only ingest via native type-stripping; corpus was seeded once via `pnpm dlx tsx`). CI pins Node 24 so `pnpm ingest` and the api dev/start scripts run normally.
- Profile bootstrap (§9.2): `ensureProfileId()` returns the localStorage `prosetype.profileId` or creates one via POST /profiles and persists it, memoizing the in-flight create so submission + stats share one; a failed create is not cached (retries next call); private-mode storage failures fall back to a session-only id.
- Result submission (§9.5) lives in the typing store, fired at run completion (before the result view appears), fire-and-forget with one retry; a module-level `submitToken` (bumped on loadNext/restart) discards stale resolutions so a late failure can't mislabel a run the user already left. Only the failure ("not saved") surfaces - in the letterbox bottom-bar left cell.
- Passage loading moved from TypingStage to StagePage, which reads the library filter from the query string (`?band`/`?theme`/`?author`): a filtered URL always (re)loads a matching passage; a bare `/` loads only when no run is live, so returning from another route keeps an in-progress passage. The active filter lives in the store and persists across Tab so a library pick "sticks".
- Library picks navigate to `/` with the filter as query params (shareable, no extra store coupling); `/library` browses bands (static 4), authors (GET /authors), and themes (GET /themes).
- web api client (`lib/api.ts`) validates responses through a structural `Parser<T>` (`{ parse(unknown): T }`) satisfied by the shared zod schemas, so the web app needs no direct `zod` dependency.
- GitHub Actions CI (`.github/workflows/ci.yml`) runs the plan §11 order as chained jobs (lint → typecheck → test → e2e) on Node 24 / pnpm 11.7.0; the `test` and `e2e` jobs attach a `postgres:16` service. `test` migrates then runs `pnpm test` with DATABASE_URL so the api integration suite executes; `e2e` also ingests the corpus (the smoke needs real passages) and installs Chromium before `pnpm test:e2e`.
- Playwright smoke (`apps/web/e2e/smoke.spec.ts`, config `apps/web/playwright.config.ts`): the config starts both servers (`api start` on :3001, `web dev` on :5173 proxying /api) and reuses already-running ones when not in CI (so it runs locally even where Node <24 can't run the api start script - start the servers by hand first). It reads the passage from `data-testid="passage"` (the board's rendered target text), types via real keyboard events at 40ms/char (keeps wpm < 350 and duration > 3s), and awaits the POST /results round-trip before reloading (a reload would abort the in-flight fire-and-forget submission).

## Phase 3

- Phase 3 item built (2026-07-09): the **Esc command palette** (first plan §10.3 backlog item), on explicit human instruction to build the first backlog item regardless of order.
- Esc now toggles a global command palette instead of directly restarting (supersedes plan §7.1 "esc → restart" and §9.3's bottom-bar "esc restart"); restart is a command inside the palette. Tab stays the fast path for next-passage; the bottom-bar hint changed to "tab next · esc commands".
- Palette open/closed state lives in its own tiny zustand store (`command/commandStore.ts`) so the typing stage can read it and bail on its own key handling, mirroring how the caps-lock flag is shared (plan silent on structure).
- Command list is a pure, DOM-free builder + filter (`command/commands.ts`, unit-tested without a renderer, since the web test env has no jsdom); the `CommandPalette` component supplies side-effecting callbacks and renders the result.
- Commands (plan silent on the set - the app has "no settings sprawl", so the palette is a launcher, not a settings surface): run-specific `Restart passage`/`Next passage` on the stage only, a `Type` nav command off-stage, always `Browse library`/`View stats`, and the four difficulty bands. Band picks navigate to `/?band=…`, reusing the existing library-pick query mechanism so there is a single filtered-load path.
- Palette search is case-insensitive substring over title + keywords (boring choice over a fuzzy matcher - the command set is small and stable).
- Esc is owned by the palette's document capture-phase handler (registered by the always-mounted `Letterbox` before the route-mounted stage handler); the stage's document handler bails whenever the palette is open so keystrokes reach the search box and Tab does not also load-next. The stage refocuses its hidden textarea when the palette closes.
- Palette overlay styling within §9.4 restraint (no borders/shadows): a dark scrim (`bg-black/75`) recedes the frame while the panel sits at `bg-stage` (a lit pool), selected row tinted `bg-tungsten/15`; motion is the existing `animate-fade-in` film cut, so reduced-motion zeroes it globally.
- Matinee (light) theme (plan §10.3): implemented as a `:root[data-theme='matinee']` override block that re-points the §9.4 `--color-*` tokens - Tailwind v4 utilities compile to `var(--color-*)`, so no utility or component changes. Theme state (`settings/theme.ts`) persists to `localStorage['prosetype.theme']` (default noir), stamps `data-theme` on `<html>`, and `initTheme()` applies it in `main.tsx` before first paint (no flash). Toggled via a palette command whose title names the destination.
- Matinee palette values (to be nudged at the feel review): stage `#e9e2d0`, bone `#2a2620`, smoke `#857c6b`, tungsten `#a9761a`, blood `#9a2e2e`, bar `#ddd4c0`. The film vignette is now `rgb(var(--color-vignette)/var(--vignette-strength))` so matinee eases it to 0.10 (dark edge reads wrong on paper); the static grain overlay stays.
- Sound profile (plan §10.3 "typewriter thock"): synthesized with the Web Audio API (band-passed noise click + short low sine body, ~40ms decay), so there are no audio assets - self-hosted like the fonts. Off by default (sound is intrusive; matches Monkeytype) and persisted to `localStorage['prosetype.sound']`; toggled via a palette command. Three variants (key/space/back) differ only in click/body frequency and gain.
- `playThock()` is a no-op when muted or where `AudioContext` is undefined (tests/SSR), so the stage input handlers call it unguarded; the shared AudioContext is created lazily on the enabling gesture (and on first keystroke) to satisfy the autoplay policy. Sound lives in the stage's DOM input handlers, not the engine, so the pure engine stays audio-free.
- Shareable result card (plan §10.3): drawn on a 1200×630 canvas at 2× (`lib/shareCard.ts`) rather than SVG→img, so the loaded web fonts render without embedding; the card is always the noir look (a brand artifact, not a theme screenshot). `shareResultCard` copies the PNG to the clipboard where supported (secure context + `ClipboardItem`) and downloads it otherwise, reporting which path it took. Attribution text is factored into `formatAttribution`, now shared by the epigraph and the card (one source of truth). Pure helpers are unit-tested; the canvas render is browser-verified (valid PNG, on-brand layout).
- Daily passage (plan §10.3): `GET /passages/daily` returns a passage chosen deterministically per UTC date - `ORDER BY md5(dateKey || '-' || id) LIMIT 1`, a stable pseudo-random pick that needs no count query and rolls over at UTC midnight (`utcDateKey`, server-computed). The static route is registered before `/passages/:id`; Fastify prioritizes static over parametric regardless, and numeric ids still resolve. Web: `loadDaily()` (sharing the load flow with `loadNext` via `loadInto`) triggered by `?daily`, reachable from a palette command; picking daily clears the active filter so a later bare Tab returns to random passages.
- Dropped from the Phase 3 batch on the human's instruction (2026-07-09): AI-generated in-style passages and Russian-original/Cyrillic passages - not built.
- Leaderboards (plan §10.3): `GET /leaderboard?passageId&limit` returns each profile's single best run (highest wpm), global or scoped to one passage (a fair same-text board). "Best per profile" uses `distinct on (profile_id) order by wpm desc` in a subquery, re-sorted into the ranking outside - written as raw `db.execute` SQL (like `listThemes`) since Drizzle has no clean distinct-on+resort builder. `db.execute` returns timestamptz as a string (the typed select paths yield a Date), so the repo re-wraps it in `new Date(...)`. Unclaimed profiles have a null `displayName`, rendered "anonymous". Limit defaults to 20, caps at 100. Reached via `/leaderboard` + a palette command; nav bar left unchanged to avoid crowding.
- Account claim (plan §10.3, email magic link): migration `0001_claim.sql` adds `profiles.email` (unique; nulls stay distinct) + `email_verified_at`, and a `claim_tokens` table (single-use, 30-min TTL). `POST /profiles/:id/claim {email}` issues a random 192-bit `base64url` token and "sends" a `${CORS_ORIGIN}/claim?token=…` link; `POST /claim/verify {token}` consumes it. Rate-limited 5/min (mailbomb deterrence).
- Claim merge is atomic (`verifyClaim` runs one `db.transaction`): if the email is unclaimed, the requesting profile becomes its owner; if another profile already owns it, that one is canonical and the requester's results are reassigned to it, then the requester's claim tokens and profile row are deleted. The profile repo touches the `results` table here (crossing its usual boundary) so the merge stays in a single transaction - a deliberate, documented exception. Display name defaults to the email local part.
- Email transport is a `Mailer` interface; the dev default (`createConsoleMailer`) logs the link via the Fastify logger - no SMTP dependency or credentials in dev scope. Wiring a real provider = implement `Mailer` and inject it in `build.ts`. Web `/claim` page does both request (email form) and verify (`?token`), adopting the returned canonical id via `setProfileId`. Full request→verify and the merge path were verified end-to-end against local Postgres; only real email delivery is unexercised (by design).
- Auto-excerpt proposer (plan §10.3): `pnpm propose <source.txt>` (`scripts/propose.ts` + pure `scripts/lib/propose.ts`) scans a PD source and ranks sentence-aligned candidate excerpts fitting §6.1 (140–450 chars, 25–80 words), each normalized via the existing `normalizeText` (windows with un-typeable chars are skipped) and scored with `computeDifficulty`/`bandForScore`. Heuristic, no AI - one candidate per starting sentence (shortest in-range window), dialogue flagged by quote presence, output as ready-to-edit YAML stubs. A curator aid; the human still chooses what to seed.
- Deployment hardening (plan §10.3 / §12 appendix): `deploy/` holds a single-host Docker Compose stack - Postgres (named volume), a one-shot `migrate` step (`service_completed_successfully` gate), the API (Node 24 runs the TS sources directly, no build stage, mirroring dev), the web built by Vite and served by Caddy which reverse-proxies `/api` (prefix preserved, since routes live under `/api/v1`), and a nightly `pg_dump` backup keeping 14 days. `SITE_ADDRESS` drives Caddy's automatic HTTPS. Verified as far as this machine allows: the Vite production build succeeds and `docker compose config` validates the whole file; images aren't built here (no Docker daemon) - `deploy/README.md` documents the deploy, seeding, restore, and that real email/off-host backups/monitoring are future work.

- Word mode (2026-07-11, `feat/word-mode`): a deliberate departure from plan §1's "one mode (quote mode)" / v1 non-goal "timed word-list modes" - added on the owner's explicit instruction so a returning user can't exhaust the 30-passage corpus. Optional, not the default: opted into via the Esc palette; the mode + word-count preset (25/50/100/200, default 200) persist in localStorage under `prosetype.mode`/`prosetype.wordCount`, mirroring the theme/sound stores. Words are a bundled ~500-word common-English list (`web/src/lib/words.ts`), lowercase and space-free so any single-space join is canonical (§6.2) and drops straight into `createEngine(text)` - zero engine changes. Sampled with replacement, matching Monkeytype. Picking a band/library/daily forces prose mode.
- Word-mode persistence: word runs are saved like prose. Since there is no stored passage, the client submits the generated text and the server recomputes stats against *that* (`POST /results` is now a `mode`-discriminated union). This means the server check proves internal consistency (client stats vs server replay of the same log+text), not that the text was "real" - inherent to a self-reported-text mode, accepted. `results` gained a nullable `passage_id`, a `mode` column, and `word_text`, guarded by a CHECK that a row is exactly one shape (migration `0002_dark_prima`). The typing store's active test became a discriminated union (`{kind:'passage'|'words'}`); the store's single `createEngine(text)` seam made this contained. Stats: word runs count in totals/averages and feed the per-key/bigram tables (clean letter data - replayed against `passageText ?? wordText`); `perAuthor` and the passage-oriented leaderboard stay prose-only (their inner joins drop word runs naturally). Result view hides the attribution epigraph and share card (a branded artifact) for word runs, showing "words · N" instead.
- Real email transport (2026-07-11): implemented `createResendMailer` (Resend, https://api.resend.com/emails) using the native `fetch` - no SDK, matching the project's minimal-dependency style. `build.ts` selects it when `RESEND_API_KEY` + `EMAIL_FROM` are set, else keeps `createConsoleMailer`. Config gates the pair (zod refine: a key with no `EMAIL_FROM` is rejected as a misconfig); both are optional so dev still logs the link with no credentials. `EMAIL_FROM` is a free-form sender ("prosetype <no-reply@prosetype.app>"), not validated as a bare email. The mailer throws on non-2xx so the claim route surfaces send failures. The compose `api` service inherits both from `.env` (env_file) - no compose change. Sending itself is still only exercised against a real Resend key (unit tests stub `fetch`).
